## MAJOR FINDING тАФ product names do NOT match, and string matching WILL fail

Appendix B: *"Match on brand + product type + size. Ignore vintage years,
formatting noise."* **That rule breaks on this invoice.**

| ILG code | Invoice description | Lightspeed product (Stowaway) |
|---|---|---|
| 122-2858 | `ALEHOUSE PREMIUM KEG` | **Alehouse Draught Lager [Keg]** |
| 122-2867 | `ALEHOUSE CRISP KEG` | **Alehouse Summer Mid [Keg]** |
| 460-1639 | `COKE NO SUGAR 1.25 LITRE` | **Coke Zero 1.25L** |
| 115-3762 | `CORONA MEXICAN 6PK BRW BX R` | Corona (Crates of 24) |
| 117-4213 | `HEAPS NORMAL QUIET XPA NON ALC` | Heaps Normal Tin (Crates of 24) |
| 450-1293 | `S.PELLEGRINO SPARKLING WATER` | San Pellegrino 500ml (Crates of 24) |
| 360-1310 | `ROOSTER ROJO TEQUILA BLANCO` | Rooster Rojo Blanco Tequila [Bottle] |
| 175-0420 | `ANTICA FORMULA` | Antica Formula Rosso Vermouth [Bottle] |

- `ALEHOUSE CRISP KEG` тЖТ **Summer Mid**. Shared tokens: "Alehouse". "Crisp" and
  "Summer Mid" have zero overlap. Brand+type matching returns nothing.
- `ALEHOUSE PREMIUM KEG` тЖТ **Draught Lager**. Same problem.
- `COKE NO SUGAR` тЖТ **Coke Zero**. Different brand names for the same product.

Worse тАФ **the same ILG product has different names per venue:**

| ILG code | Stowaway | Harry Gatos |
|---|---|---|
| 122-2858 | Alehouse **Draught Lager** [Keg] | Alehouse **Premium Lager** [Keg] |

Appendix A documents that ProductIDs are venue-specific. Names are too, and they
diverge from each other AND from the supplier's description.

### Design consequence

**Product resolution must key on the supplier's item code, not the description.**
ILG gives a code on every line (`122-2858`, `460-1639`, ...). IWI does too
(`GHEMILL-24`). That code тЖТ ProductID mapping is a table that must be built and
maintained per venue тАФ it cannot be inferred, and fuzzy matching on descriptions
will silently mis-map `ALEHOUSE CRISP` onto the wrong keg.

This is a hard blocker for automated receiving and was invisible until a real
invoice was compared against a real PO.

## Cost-basis note (not an error)

The Purchase/receive screen shows the price per **ordering unit** (crate/box):
Corona `$61.71` for 1 crate, total `$61.71`. Back Office Cost Price is per
**stock unit** (Rule 9: beer tins per-can, $2тАУ3). Both true, different fields:

```
Corona   TOT 61.71 / 24 = $2.57 per bottle   <- BO basis
Heaps    TOT 64.07 / 24 = $2.67 per tin      <- BO basis
```

Confirmed by Zak: Stowaway/HG stock **singles**, not 4- or 6-packs тАФ so the
`(Crates of 24)` naming and `TOT / (qty ├Ч pack_size)` agree. This is also why
ILG's LUC (per 6-pack for Corona, per 4-pack for Heaps) is unusable here.
